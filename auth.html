Authentication

Bearer Token

All endpoints require a Bearer token in the Authorization header:

Authorization: Bearer rise_abc123def456...

Tokens are validated against the database using SHA-256 hashing. If the token is invalid, expired, or inactive, the API returns a 401 Unauthorized response.

Token Exchange (Session Tokens)

For environments where storing a permanent API key is a security risk (e.g., client-side applications), you can use the token exchange flow:

  1. Generate an API key with auth_mode = "exchange"
  2. POST the key to /api/v1/auth/session to get a temporary session token
  3. Use the session token as a Bearer token (expires in 30 minutes)
curl -X POST https://your-rise.com/api/v1/auth/session \
  -H "Content-Type: application/json" \
  -d '{"api_key": "rise_exchange_key..."}'

Response:
{"success": true, "data": {"session_token": "rise_sess_...", "expires_in": 1800}}

Scopes

Each token has a set of scopes that define what it can do. Scopes follow the format:

FormatExampleDescription
{module}:{action}tasks:readSpecific action on a module
{module}:*tasks:*All actions on a module
**Full access (admin only)

Rate Limiting

Rate limiting is enforced using a sliding window of 1 minute:

User TypeRequests per Minute
Admin300
Staff60
Client60

When exceeded, the API returns 429 Rate limited.

Authorization Checks

RISE API implements three levels of access control:

  • Scope check — The token must have the required scope for the endpoint
  • Ownership check — Staff users can only access their own resources (configurable via requireOwnership())
  • Client restriction — Client users can only access resources belonging to their client account

CORS

CORS headers are applied when origins are configured in Settings → API Keys → CORS Origins.

Access-Control-Allow-Origin: https://your-app.com
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
